# 苍穹外卖项目 — 技术栈解析与面试指南

---

## 一、项目概况

| 项 | 说明 |
|---|------|
| 项目名称 | 苍穹外卖（Sky Take-Out） |
| 项目类型 | 外卖点餐平台（B2C） |
| 架构模式 | 前后端分离，单体后端 + 微信小程序 |
| 后端技术 | Spring Boot 2.7.3 + MyBatis 2.2.0 + MySQL 8 + Redis 6 |
| JDK 版本 | 11 |
| 项目结构 | 多模块 Maven 工程（sky-common / sky-pojo / sky-server） |

**业务定位：**
- **管理端**：商家后台（员工管理、菜品分类管理、菜品管理、套餐管理、订单管理、数据统计）
- **用户端**：微信小程序（用户浏览菜品、下单、支付、查看订单）

## 二、技术栈全景解析

### 2.1 核心框架

#### Spring Boot 2.7.3
- 自动配置：通过 `@SpringBootApplication` 启动
- profile 管理：`application.yml` 占位 + `application-dev.yml` 注入，敏感信息抽离
- `allow-circular-references: true`：允许循环依赖（MyBatis 多模块场景）

> **面试考点**：`@SpringBootApplication` 是 `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan` 的复合注解

#### MyBatis 2.2.0 + PageHelper 1.3.0
- **注解 SQL**：简单查询用 `@Select` / `@Insert` / `@Update` / `@Delete`
- **XML SQL**：复杂查询（分页、动态条件、批量插入）用 XML mapper
- **驼峰映射**：`map-underscore-to-camel-case: true` 自动转换
- **PageHelper**：物理分页插件

> **面试考点**：`#{}` vs `${}` 的区别、MyBatis 一二级缓存、PageHelper 原理（拦截器 + ThreadLocal）

#### Alibaba Druid 1.2.1
- 数据库连接池，自带 SQL 监控和防火墙
- 通过 `spring.datasource.druid.*` 配置

> **面试考点**：连接池参数（initial-size、max-active、min-idle）、Druid vs HikariCP

### 2.2 安全认证

#### JWT（JSON Web Token）— 双端认证
```
管理端：密钥=itcast，过期=24h，Header=token，载荷={empId}
用户端：密钥=ithuang，过期=2h，Header=authentication，载荷={userId}
```

**认证链路**：
1. 登录成功 → `JwtUtil.createJWT()` 生成 token → 返回前端
2. 前端每次请求 header 携带 token
3. 拦截器 `JwtTokenAdminInterceptor` / `JwtTokenUserInterceptor` 解析验证
4. 解析出用户 ID → `BaseContext.setCurrentId()` 存入 ThreadLocal
5. 请求结束从 ThreadLocal 读用户 ID

> **面试考点**：JWT 三段结构（header.payload.signature）、HS256 对称加密、token 过期刷新策略、JWT 如何防篡改（签名验证）、ThreadLocal 内存泄漏问题

### 2.3 公共模块设计（sky-common）

#### 统一返回格式 Result\<T\>
```java
Result.success(data)      → { "code": 1, "data": {...} }
Result.error("错误信息")  → { "code": 0, "msg": "错误信息" }
```

#### 全局异常处理 GlobalExceptionHandler
- `@RestControllerAdvice` 拦截所有 Controller 异常
- 区分处理：业务异常（`BaseException` 子类）返回中文提示、SQL 约束冲突（`SQLIntegrityConstraintViolationException`）解析唯一约束字段
- 所有异常统一包装为 `Result.error()`

> **面试考点**：`@RestControllerAdvice` vs `@ControllerAdvice`、Spring 异常处理链路（`HandlerExceptionResolver` → `@ExceptionHandler`）

#### Jackson 序列化配置
- `JacksonObjectMapper` 扩展 `ObjectMapper`
- 自定义 `LocalDateTime` → `"yyyy-MM-dd HH:mm"`、`LocalDate` → `"yyyy-MM-dd"`
- 全局注册到 `MappingJackson2HttpMessageConverter`（优先级最高）
- JSON 属性命名策略：`SNAKE_CASE`（全局下划线）

> **面试考点**：Jackson 自定义序列化方案（`@JsonFormat` vs 全局 `ObjectMapper`）、消息转换器注册顺序

#### ThreadLocal 用户上下文 BaseContext
- 在当前线程传递用户/员工 ID
- 拦截器 set → 业务层 get → 请求结束 remove（防止内存泄漏）

> **面试考点**：ThreadLocal 底层（Thread.threadLocals 数组）、弱引用 Entry、key 为 null 的 value 无法回收 → `remove()` 的必要性

### 2.4 AOP 自动填充

```
@AutoFill(INSERT)  → 自动填入 createTime, createUser, updateTime, updateUser
@AutoFill(UPDATE)  → 自动填入 updateTime, updateUser
```

**实现**：`AutoFillAspect` 切面拦截 `com.sky.mapper.*.*(..)` 且带 `@AutoFill` 注解的方法，通过反射调用实体 setter 方法。

> **面试考点**：AOP 核心概念（JoinPoint、Pointcut、Advice、Aspect、Weaving）、JDK 动态代理 vs CGLIB、`execution` 表达式语法

### 2.5 定时任务（Spring Task）

| 任务 | cron | 功能 |
|------|------|------|
| `OrderTask.processTimeoutOrder()` | `0 * * * * ?` | 每分钟扫描超时未付订单（15分钟），自动取消 |
| `OrderTask.processDeliveryOrder()` | `0 0 1 * * ?` | 每天凌晨1点，将配送中的订单标记为已完成 |

通过 `@EnableScheduling` + `@Scheduled(cron = "...")` 实现。

> **面试考点**：cron 表达式语法、`cron()` vs `fixedRate` / `fixedDelay`、分布式定时任务方案对比（XXL-JOB、Elastic-Job、Quartz）

### 2.6 WebSocket 实时推送

- `@ServerEndpoint("/ws/{sid}")` 注解式 WebSocket
- `WebSocketServer` 维护 `Map<String, Session>` 管理客户端连接
- `WebSocketTask` 定时广播消息（用于向商家推送新订单通知）

> **面试考点**：WebSocket 握手过程（HTTP Upgrade 101）、心跳保活（ping/pong）、Nginx 反向代理 WebSocket 配置、WebSocket vs SSE vs 轮询

### 2.7 Redis 缓存

- Spring Data Redis + Lettuce 客户端
- 自定义 `RedisTemplate`：Key 用 `StringRedisSerializer`（防止乱码）
- 通过 `@EnableCaching` 开启 Spring Cache 抽象
- 菜品数据使用 `@Cacheable` / `@CacheEvict` 缓存

> **面试考点**：缓存穿透/击穿/雪崩、`RedisTemplate` vs `StringRedisTemplate`、序列化方案对比（JDK vs JSON vs String）、缓存与数据库一致性策略（先删缓存还是先更新库）

### 2.8 文件上传 — 阿里云 OSS

- `AliOssUtil` 封装 OSS SDK → `putObject()` 上传 byte[]
- 返回文件公网 URL
- 通过 `@ConditionalOnMissingBean` 条件注册组件
- 前端请求 → `CommonController` 接收 `MultipartFile` → service 调用阿里云 OSS

> **面试考点**：大文件上传方案（分片上传、断点续传）、临时签名 URL、CDN 加速、`@ConditionalOnMissingBean` 作用

### 2.9 微信生态集成

#### 微信登录（小程序）
```
小程序端 wx.login() → code
后端 UserServiceImpl.wxLogin(code) → 发请求到微信接口
https://api.weixin.qq.com/sns/jscode2session?appid=xxx&secret=xxx&js_code=code
→ 返回 { openid, session_key }
→ 查询 user 表，新用户自动注册
→ 生成 JWT token 返回前端
```

#### 微信支付 V3 API
- `WeChatPayUtil` 使用微信官方 SDK（wechatpay-apache-httpclient）
- 自动处理：SHA256withRSA 签名、平台证书验证、请求/响应签名校验
- JSAPI 下单 → 获取 prepay_id → 二次签名 → 小程序调起支付
- 退款：调用退款接口，金额按分计算

> **面试考点**：小程序登录流程（code → openid → token）、HTTPS + 证书双向认证、用户数据解密、支付回调幂等性处理

### 2.10 API 文档 — Knife4j / Swagger

- 基于 Springfox 3.0.0
- 分组文档：`/doc.html` 访问，分为"管理端接口"和"用户端接口"两组
- `@Api`、`@ApiOperation` 注解标记

### 2.11 POI Excel 导出

- Apache POI 3.16 生成 `.xlsx` 格式报表
- 用于数据统计模块的营业额/订单报表导出

> **面试考点**：POI 写入 Excel 的内存问题（SXSSFWorkbook 流式写入）、导出数据量大的优化策略

## 三、项目架构亮点总结

```
┌──────────────────────────────────────────────────┐
│           Interceptor (JWT 认证层)                │
│   admin → JwtTokenAdminInterceptor               │
│   user  → JwtTokenUserInterceptor                │
├──────────────────────────────────────────────────┤
│           Controller (接口层)                     │
│   admin/ (9个)  |  user/ (8个)  |  notify/ (1个) │
├──────────────────────────────────────────────────┤
│           Service (业务逻辑层)                     │
│         10个接口 + 10个实现类                       │
├──────────────────────────────────────────────────┤
│           Mapper (数据访问层)                      │
│         11个接口 + 11个 XML                        │
├──────────────────────────────────────────────────┤
│           MySQL + Redis + OSS                     │
└──────────────────────────────────────────────────┘

横切关注点：
├── @AutoFill AOP      → 公共字段自动填充
├── ThreadLocal         → 用户上下文传递
├── GlobalExceptionHandler → 统一异常处理
├── JacksonObjectMapper → 统一 JSON 序列化
├── @Scheduled Tasks    → 订单状态定时处理
└── WebSocket           → 实时消息推送
```

**设计模式体现：**
- **模板方法模式**：统一 `Result<T>` 返回格式
- **策略模式**：`OperationType` 枚举 + AOP 分发
- **拦截器模式**：JWT 认证
- **门面模式**：Service 层封装复杂业务逻辑

## 四、面试高频问题

### 基础部分

**1. 这个项目的整体架构是怎样的？你负责了哪些模块？**

建议回答思路：前后端分离架构，后端单体 Spring Boot 多模块工程。服务管理端和用户端两套 API（通过不同 Controller 包和拦截器路径区分 `/admin/**` 和 `/user/**`）。介绍自己深入参与的模块。

---

**2. 为什么要分 sky-common 和 sky-pojo 两个公共模块？**

- **sky-common**：与业务无直接耦合的通用组件（工具类、异常、常量、结果封装）
- **sky-pojo**：与数据模型直接相关的类（Entity、DTO、VO）
- 好处：可复用、职责清晰、按需依赖（server 引 common + pojo，common 不依赖 pojo）

---

**3. 项目中如何做登录认证的？**

JWT 双端认证：
- 管理端（admin）：Header 名 `token`，密钥 `itcast`，24h 过期
- 用户端（user）：Header 名 `authentication`，密钥 `ithuang`，2h 过期
- 自定义拦截器实现（`HandlerInterceptor`），在 `WebMvcConfiguration` 中注册，排除登录接口

---

**4. ThreadLocal 是什么？为什么最后要 remove？**

- 线程本地变量，每个线程持有独立副本
- 这个项目中 `BaseContext` 用 ThreadLocal 存储当前请求的用户 ID，方便在 Service/Mapper 层任何地方获取
- 使用 **Tomcat 线程池**，线程是复用的，请求结束后如果不 remove，下一个请求可能读到上一个请求的用户数据，造成数据串错
- 还有**内存泄漏风险**：ThreadLocalMap 的 Entry 中 key 是弱引用但 value 是强引用，key 被 GC 后 value 无法回收

---

**5. AOP 自动填充是怎么实现的？**

1. 定义 `@AutoFill` 注解，标记在 Mapper 方法上
2. `AutoFillAspect` 通过 `@Aspect` 切面拦截带 `@AutoFill` 的方法
3. 根据注解的 `OperationType`（INSERT/UPDATE），反射调用实体的 setter
4. 从 `BaseContext.getCurrentId()` 取当前操作用户 ID

---

**6. `#{}` 和 `${}` 的区别是什么？**

- `#{}`：预编译占位符，生成 `?`，**防止 SQL 注入**，MyBatis 自动类型处理
- `${}`：字符串拼接，直接拼到 SQL 中，存在注入风险，仅用于动态表名/列名等非值场景

---

### 进阶部分

**7. 订单超时自动取消是怎么做的？**

- Spring `@Scheduled(cron = "0 * * * * ?")` 每分钟执行一次
- 查询 `status=待支付` 且 `order_time < 当前时间-15分钟` 的订单
- 批量更新状态为"已取消"，记录取消原因和取消时间

> 追问：这在分布式环境下有什么问题？如何解决？
> 答：多实例部署时每个实例都会执行，造成数据重复处理。解决方案：XXL-JOB 的任务分片、Redisson 分布式锁、单实例执行

---

**8. 微信小程序登录的完整流程？**

1. 小程序前端调用 `wx.login()` 获取临时 `code`
2. 前端将 `code` 传给后端 `/user/user/login`
3. 后端用 `code` + `appid` + `secret` 请求微信 `jscode2session` 接口
4. 微信返回 `{ openid, session_key }`（session_key 不能给前端）
5. 根据 openid 查数据库，新用户自动注册
6. 生成 JWT token 返回前端
7. 前端拿到 token 存入 `wx.setStorageSync('token', xxx)`，后续请求携带

---

**9. 如果线上出现超卖（库存为负），你会怎么排查和解决？**

排查方向：
1. 检查下单逻辑是否有库存校验
2. 检查是否有并发下单场景（同一商品多人同时买）
3. 数据库隔离级别是否足够

解决：
- **乐观锁**：`update dish set stock = stock - 1, version = version + 1 where id = #{id} and stock >= #{quantity} and version = #{version}`，更新行数为 0 则重试或提示
- **Redis 分布式锁**：对商品 ID 加锁再扣库存
- **Redis 预扣库存**：秒杀场景用 Redis decr 原子操作

---

**10. `@Transactional` 什么情况下会失效？**

1. 方法不是 public 的（CGLIB 代理无法拦截）
2. 同一个类内部调用（this.xxx() 不经过代理对象）
3. 异常被 catch 吞掉了，框架感知不到
4. 抛了 checked Exception（默认只回滚 RuntimeException 和 Error）
5. 数据库引擎不支持事务（如 MyISAM）

---

**11. Redis 在你们项目中用在了哪些场景？**

- **缓存菜品分类**：`@Cacheable` 缓存 `category` 表
- **缓存菜品**：`@CacheEvict` 在新增/修改/删除时清除缓存
- **店铺营业状态**：`RedisTemplate` 存取 `SHOP_STATUS` key
- 如果进一步实现：可以缓存 JWT token 实现主动失效、缓存用户 session

---

**12. 全局异常处理怎么做的？**

- `@RestControllerAdvice` 类 `GlobalExceptionHandler`
- `@ExceptionHandler(BaseException.class)` 处理业务异常
- `@ExceptionHandler(SQLIntegrityConstraintViolationException.class)` 处理唯一约束冲突，解析 "Duplicate entry 'xxx' for key 'yyy'" 字符串，提取重复字段返回友好提示
- 最终统一包装为 `Result.error(msg)` 返回

---

**13. 如何保证支付回调的幂等性？**

1. 收到微信支付回调后，先根据 `transaction_id`（微信订单号）或 `out_trade_no`（商户订单号）查询订单状态
2. 如果订单已经是"已支付"，直接返回成功，不重复处理
3. 如果尚未处理，更新订单状态 + 记录支付日志 → **这两个操作在同一事务中**，保证一致性
4. 微信回调可能重复发（5次内），必须处理重复通知

---

**14. 项目中有哪些可以优化的地方？**

- **缓存策略**：加入缓存预热（启动时加载热门菜品）、布隆过滤器防穿透
- **SQL 优化**：分析慢查询，加索引（order_time、status、user_id 联合索引）
- **接口限流**：用 Redis + 滑动窗口对登录/下单接口限流
- **分布式改造**：订单服务独立部署，用 Seata 做分布式事务
- **消息队列**：订单创建后发 MQ 消息异步处理短信通知、库存扣减
- **读写分离**：报表查询走从库

**16. PageHelper 分页原理是什么？**

PageHelper 是 MyBatis 的分页插件：
1. 调用 `PageHelper.startPage(pageNum, pageSize)` 时，将分页参数存入 ThreadLocal
2. MyBatis 执行 SQL 前，PageHelper 的拦截器从 ThreadLocal 取出分页参数
3. 包装原始 SQL，在后面追加 `LIMIT offset, limit`
4. 同时执行 `COUNT` 查询获取总记录数
5. 最终返回 `Page` 对象（继承 `ArrayList`，包含 total 等分页信息）

---

**17. Jackson 的 SNAKE_CASE 策略是什么？为什么不直接用 @JsonProperty？**

- `SNAKE_CASE`：Java 的驼峰命名（`userId`）序列化为 JSON 的蛇形命名（`user_id`），全局生效
- 项目中配置了 `spring.jackson.property-naming-strategy: SNAKE_CASE`，全局设置
- 比 `@JsonProperty` 的好处：不用在每个字段上标注，约定优于配置
- 但项目中并不是全部走 SNAKE_CASE 策略，因为前端传参用的是 `@RequestBody`，会自动反序列化回来
- 关键：这是在 `application.yml` 全局配置的，会影响所有 Controller 的请求/响应
